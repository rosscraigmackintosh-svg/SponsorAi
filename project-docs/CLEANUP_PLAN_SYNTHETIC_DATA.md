## Identification Strategy

All synthetic properties were created on 2026-02-28 (the seed date from
master_schema.sql). British GT data is inserted on 2026-03-13 or later.

Primary filter for synthetic properties:

```sql
created_at::date = '2026-02-28'
```

This is the most reliable discriminator. It is independent of bio text, name
format, or property_type.

Secondary confirmation: all synthetic rows also have `sport IS NULL` and
`slug IS NULL`. Both will remain NULL until Phase 2 backfill or explicit update.

Cross-check before running cleanup (STEP 0 below): verify the count of rows
matching the filter equals 120 exactly.

## Safe Deletion Order

CASCADE constraints handle most of the chain automatically when properties are
deleted. However, the explicit deletion order below is recommended for large
datasets to avoid long lock chains and to give visibility into each step.

### Step 0: Identify and verify -- run before any deletion

```sql
-- Confirm synthetic property count before cleanup
SELECT
  property_type,
  COUNT(*) AS cnt,
  MIN(created_at)::date AS earliest,
  MAX(created_at)::date AS latest
FROM public.properties
WHERE created_at::date = '2026-02-28'
GROUP BY property_type
ORDER BY property_type;

-- Expected result:
-- driver: 50, 2026-02-28, 2026-02-28
-- team:   30, 2026-02-28, 2026-02-28
-- series: 20, 2026-02-28, 2026-02-28
-- event:  20, 2026-02-28, 2026-02-28

-- Confirm British GT data exists (would fail if seed not yet run)
SELECT property_type, COUNT(*)
FROM public.properties
WHERE sport = 'motorsport'
GROUP BY property_type;
```

### Step 1: Delete property_relationships

Delete explicitly first -- faster than waiting for CASCADE on a 50-row table,
and gives a clear confirmation before the large deletions.

```sql
DELETE FROM public.property_relationships
WHERE from_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
)
OR to_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
);
-- Expected: 50 rows deleted
```

### Step 2: Delete FanScore data

```sql
DELETE FROM public.fanscore_windows
WHERE property_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
);
-- Expected: 1,080 rows deleted

DELETE FROM public.fanscore_daily
WHERE property_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
);
-- Expected: 21,720 rows deleted
```

### Step 3: Delete property_platform_daily_metrics

```sql
DELETE FROM public.property_platform_daily_metrics
WHERE property_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
);
-- Expected: 43,259 rows deleted
```

### Step 4: Delete raw post metrics and follower data

These are linked via accounts, not directly via properties. Collect account
IDs first, then delete.

```sql
-- Raw post metrics (via post -> account -> property)
DELETE FROM public.raw_post_daily_metrics
WHERE post_id IN (
  SELECT rp.id FROM public.raw_posts rp
  JOIN public.accounts a ON a.id = rp.account_id
  JOIN public.properties p ON p.id = a.property_id
  WHERE p.created_at::date = '2026-02-28'
);
-- Expected: ~396,862 rows deleted (this is the heaviest step)

-- Follower snapshots (via account -> property)
DELETE FROM public.raw_account_followers
WHERE account_id IN (
  SELECT a.id FROM public.accounts a
  JOIN public.properties p ON p.id = a.property_id
  WHERE p.created_at::date = '2026-02-28'
);
-- Expected: 43,259 rows deleted

-- Raw posts (via account -> property)
DELETE FROM public.raw_posts
WHERE account_id IN (
  SELECT a.id FROM public.accounts a
  JOIN public.properties p ON p.id = a.property_id
  WHERE p.created_at::date = '2026-02-28'
);
-- Expected: 29,415 rows deleted
```

### Step 5: Delete accounts

```sql
DELETE FROM public.accounts
WHERE property_id IN (
  SELECT id FROM public.properties WHERE created_at::date = '2026-02-28'
);
-- Expected: 239 rows deleted
```

### Step 6: Delete synthetic properties (final step)

```sql
DELETE FROM public.properties
WHERE created_at::date = '2026-02-28';
-- Expected: 120 rows deleted
```

### Step 7: Post-cleanup verification

```sql
-- Confirm nothing remains from the synthetic seed
SELECT COUNT(*) AS synthetic_remaining
FROM public.properties
WHERE created_at::date = '2026-02-28';
-- Expected: 0

-- Confirm British GT data is intact
SELECT property_type, COUNT(*) AS cnt
FROM public.properties
WHERE sport = 'motorsport'
GROUP BY property_type
ORDER BY property_type;
-- Expected: athlete=20, event=7, governing_body=1, series=1, team=12, venue=6

-- Confirm row counts in dependent tables
SELECT 'accounts' AS tbl, COUNT(*) FROM accounts
UNION ALL SELECT 'fanscore_daily', COUNT(*) FROM fanscore_daily
UNION ALL SELECT 'fanscore_windows', COUNT(*) FROM fanscore_windows
UNION ALL SELECT 'property_relationships', COUNT(*) FROM property_relationships;
-- After cleanup: accounts=0, fanscore_daily=0, fanscore_windows=0
-- property_relationships should equal the 67 British GT relationships
```

"Global Touring Championship", "Alex Storm") map to real entities and none should

(no aliases or metrics have been added to synthetic data). No action needed on

## Notes

**Volume warning:** Step 4 deletes ~470k rows across three tables. On Supabase
free/pro tier this should complete within seconds, but the operation should not
be interrupted once started. Run during low-traffic period if possible.

**No cascade shortcut:** Although a single `DELETE FROM properties WHERE
created_at::date = '2026-02-28'` would cascade to everything, the explicit
step-by-step order above is recommended for the first cleanup run. It provides
row-count confirmation at each step and avoids a single enormous lock.

**fanscore_models is not affected:** The one fanscore_models row (v1.0) has no
created_at dependency on synthetic properties. It must not be deleted.

**RLS not a factor:** Cleanup SQL must be run as a service_role or superuser
operation (via Supabase dashboard or MCP apply_migration tool). The anon_read
policies are SELECT-only and do not affect DELETE operations.
